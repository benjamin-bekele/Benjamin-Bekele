## 1) Deterministic scoring: fixed consistency vs adaptive recalibration (but still reproducible)
**Best practice:** prioritize **consistency in the scoring formula**, but allow **adaptation via versioned recalibration**.

- **Make the scoring pipeline deterministic for identical inputs** by keeping:
  - model weights frozen for a given “scoring version”
  - the mapping from features → score using fixed parameters
  - the same preprocessing rules (tokenization, deduping, time windows)

- **Allow recalibration without breaking reproducibility** by using **versioned updates**:
  - recalibration produces a new “benchmark/model version” (e.g., v1.3 → v1.4)
  - the system stores which version was used for each score
  - identical historical inputs can be re-scored to the *same* published result by using the recorded version

So, adapt over time by **rolling out new score versions**, not by changing the live scoring parameters invisibly.

## 2) Multi-modal identity score: which modality should dominate, and how to set weights
There’s no universal ordering, but a common, defensible approach is:

- **Highest influence for “identity definition” → Textual messaging and semantics**
  - because identity is often expressed directly in language: positioning, claims, tone, themes.
- **Second-highest → Visual aesthetics (when brand is visually distinctive)**
  - useful for premium/luxury, design systems, recognition cues, and style coherence.
- **Lower to medium baseline → Platform behavioral signals**
  - these are more about *execution and strategy* (and detecting anomalies), but they’re more contingent on platform norms and can be easily gamed.

A practical weighting strategy:
- Use **context-dependent weighting**:
  - if a brand is visually driven, raise image/video weight
  - if textual positioning is central, raise text weight
- And set weights as **user-configurable with sensible defaults**:
  - Provide default weights derived from internal validation or “benchmark-like” training on labeled examples.
  - Let users override for their domain/scope, but keep the choice **versioned** and explainable.

**Recommendation:** start with weights inferred from large-scale evaluation (benchmarks / labeled ground truth) for good defaults, but expose user controls for domain fit and governance.

## 3) Competitor sets: dynamic vs static, and re-evaluation frequency
**Recommendation:** prefer **mostly static sets** for longitudinal comparability, with **controlled updates**.

- **Keep sets static** within a scoring period/version so that trend lines mean something (same reference group).
- **Update dynamically via “review cycles”**:
  - detect new entrants, remove obvious no-longer-relevant competitors, and adjust messaging similarity
  - apply changes only at defined checkpoints, producing a new “competitor-set version”

**How frequently to re-evaluate?**
- If you need meaningful trends but brands/markets shift quickly: **monthly or quarterly**
- For slower-moving categories: **quarterly to yearly**
- A good compromise is: **continuous detection**, but **scheduled adoption** (e.g., “rebuild candidate set daily/weekly, approve changes quarterly”).

This gives you:
- comparability (users can track identity score over consistent references)
- realism (the reference group can evolve when the market truly changes)
